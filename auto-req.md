# auto-req — 需求对齐 Prompt 模板

在 Agent 写任何代码之前，消除需求中的所有模糊地带。产出一份可直接进入开发阶段的结构化需求文档。

---

## 模式 A：穷尽式追问（适合核心需求，你有明确想法但需要充分沟通）

### 启动 Prompt

```
你现在处于 Plan Mode。在你开始写任何代码、做任何技术规划之前，我必须先确保你完全理解这个需求。

需求描述：
---
[在此处粘贴你的自然语言需求描述]
---

你的任务：
1. 仔细阅读上述需求描述
2. 列出所有你仍然不确定、需要我澄清的问题。包括但不限于：
   - 需求中未明确约定的行为
   - 存在多种合理实现方式的决策点
   - 边界条件和异常场景
   - 与现有系统/模块的交互假设
   - 性能、安全、并发等非功能性约束
   - 术语定义（如果需求中使用了领域特定词汇）
3. 将问题按重要性排序，最重要的放在前面

规则：
- 不要开始规划技术方案
- 不要写任何代码
- 不要假设"应该是这样"——有任何不确定就问
```

### 追问轮次 Prompt

```
以下是我对你问题的回答：
---
[逐一回答]
---

基于我的回答，你产生了新的不确定之处吗？如果有，继续问。如果没有，进入下一步：根据我们所有的问答，输出一份完整的结构化需求文档。
```

### 终止条件

当 Agent 提出的问题开始变得琐碎、重复，或已经进入"如果用户没特别要求就按默认处理"的范畴时，终止追问，进入产出阶段。

---

## 模式 B：方案生成 + 人类筛选（适合外围需求，你不在乎具体实现方式）

### 启动 Prompt

```
你现在处于 Plan Mode。在你开始写任何代码之前，先帮我补全需求中的缺失决策。

需求描述：
---
[在此处粘贴你的自然语言需求描述]
---

你的任务：
1. 分析上述需求描述，找出所有模糊、未定义、或有多种合理解读的部分
2. 对每个模糊点，基于你的世界知识和主流产品的最佳实践，给出 Top-3 方案（如果少于 3 种合理方案，列出全部）
3. 每个方案包含：
   - 方案描述（一句话）
   - 推荐理由（为什么这是合理的）
   - 潜在缺点（诚实列出）
   - 参考案例（哪些知名产品采用了类似方案）

格式示例：
## 模糊点 1：用户注册方式
- 方案 A：邮箱 + 密码注册，邮件验证链接激活
  - 推荐：SaaS 主流做法，实现简单，spam 风险可控
  - 缺点：用户可能漏看邮件，激活率损失
  - 参考：Notion, Linear
- 方案 B：手机号 + 短信验证码
  - 推荐：国内用户习惯，激活率高
  - 缺点：短信成本，国际用户覆盖差
  - 参考：微信, 抖音
- 方案 C：第三方 OAuth（Google/GitHub）优先
  - 推荐：减少密码管理负担，适合开发者工具
  - 缺点：依赖第三方可用性，部分用户隐私顾虑
  - 参考：Vercel, Supabase

规则：
- 不要写代码
- 不要替我做决定——只呈现选项和利弊
- 如果某个决策点需要搜索最新信息才能给出合理建议，请说明并给出搜索关键词
```

### 人类决策后 Prompt

```
我的选择如下：
---
- 模糊点 1：选方案 A，但邮箱验证改为可选（注册即可使用基础功能）
- 模糊点 2：选你推荐的方案 C
- 模糊点 3：...
---

基于以上决策 + 原始需求，输出一份完整的结构化需求文档。
```

---

## 结构化需求文档输出格式

无论用模式 A 还是 B，最终产出应包含以下章节：

```markdown
# 需求文档：[功能/模块名称]

## 1. 背景与目标
- 为什么要做这个功能？解决什么问题？
- 成功的标准是什么？

## 2. 用户故事 / 用例
- 作为 [角色]，我想要 [目标]，以便 [收益]
- 覆盖主流程和已知分支

## 3. 功能范围
### In Scope
- 明确列出必须实现的功能点
### Out of Scope
- 明确列出本次不做的功能点（防止范围蔓延）

## 4. 详细行为规格
- 每个功能点的输入、处理逻辑、输出
- 状态转换规则（如有）
- 错误场景及预期行为

## 5. 边界条件与约束
- 数据量上限、并发用户数、响应时间要求
- 兼容性要求（浏览器、操作系统、API 版本）
- 安全与合规约束

## 6. 接口与依赖
- 需要调用哪些内部/外部接口
- 对现有模块的影响
- 数据库变更（如有）

## 7. 验收标准
- 可验证的通过/失败条件
- 测试重点提示（给后续 TPDD 阶段参考）

## 8. 待确认事项（可选）
- 如果仍有无法在本阶段确定的事项，列出并标注负责人和截止时间
```

---

## 混合使用建议

| 需求类型 | 推荐模式 | 原因 |
|---------|---------|------|
| 核心业务逻辑 | A（穷尽式追问） | 偏差代价高，必须充分对齐 |
| 通用 CRUD/辅助功能 | B（方案筛选） | 标准做法成熟，Agent 先验知识足够 |
| 创新/无参照的功能 | A + B 结合 | 先用 A 摸清核心假设，再用 B 补全外围 |

**典型流程：**
1. 核心流程用模式 A 对齐 10-20 分钟
2. 外围细节用模式 B 让 Agent 补全方案
3. 合并为一份需求文档，进入 TPDD 阶段
