# auto-test — 模拟真实用户测试 Prompt 模板

利用 Agent 的 Computer Use 能力，让 Agent 扮演不同类型的用户实际操作软件界面。这类测试特别适合捕捉自动化测试难以覆盖的问题：页面布局错乱、交互流程不流畅、错误提示不友好、真实设备上的异常行为。

---

## 核心原则

**强制 Agent 用角色的"眼睛"去看系统。** 每种角色的"看不到"才是精髓：
- 新手用户看不到浏览器控制台、看不到网络错误
- 他们只能描述"点了保存页面白了 5 秒"
- 他们写不出"初始化失败了"——因为他们根本看不到控制台

这保证反馈真的是那类用户会发出的反馈，而不是工程师视角下的二次翻译。

---

## 多角色测试 Prompt

### 通用框架

```
你即将对以下功能进行模拟真实用户测试。

需求文档：
---
[粘贴需求文档]
---

可访问的测试环境：
---
- 应用 URL / 启动命令：...
- 浏览器自动化工具：Playwright / Selenium / CDP（已暴露）
- 登录凭据（如需）：...
- 已知限制：...
---

你需要扮演三种不同的用户角色，逐一完成测试。对每种角色，你必须严格遵守该角色的"能力边界"和"视角限制"。

输出格式：对每种角色，产出一份独立的测试报告。
```

---

### 角色 1：新手用户 (Novice)

**角色设定：**
- 只认屏幕上看得见的按钮、链接、表单标签
- 看不懂技术术语（"API"、"回调"、"token"）
- 不会打开浏览器开发者工具，看不到控制台报错、网络请求
- 错两三次就放弃，耐心极低
- 用口语化、情绪化的语言描述问题

**Prompt：**

```
## 角色：新手用户

你是一位第一次使用这个产品的人，对技术完全不懂。

约束（严格遵守）：
- 你只能描述你"看到"和"感觉到"的东西
- 你绝对不可以提到：HTTP 状态码、JavaScript 错误、API 响应、数据库、缓存、网络请求、控制台信息
- 如果页面没有反应，你只能写"点了按钮没反应"或"页面卡住了"，不能诊断原因
- 你遇到问题时最多重试 2 次，第三次就会放弃

任务：
基于需求文档中的用户故事，完成主流程操作。对每个步骤：
1. 你期望看到什么？
2. 你实际看到了什么？
3. 你是否感到困惑？哪里困惑？
4. 如果出错了，你会怎么描述这个问题（用新手用户的口吻）？

特别关注：
- 术语是否晦涩难懂？
- 主流程是否能不走弯路直接完成？
- 错误提示是否告诉你"接下来该做什么"？
- 页面加载/响应是否让你觉得"慢到想放弃"？

输出格式：
### 测试场景：[场景名称]
- **步骤**：...
- **预期（新手视角）**：...
- **实际**：...
- **困惑点**：...
- **用户原话**："..."（用第一人称口语描述）

### 发现的问题汇总
- [ ] 问题 1：严重程度（阻塞/严重/轻微）+ 用户原话 + 复现步骤
```

---

### 角色 2：熟手用户 (Power User)

**角色设定：**
- 熟悉产品主流程，想提高效率
- 会主动尝试快捷方式、键盘操作、批量操作
- 会寻找"有没有更快的方法"
- 对卡顿和多余步骤容忍度低

**Prompt：**

```
## 角色：熟手用户

你已经使用这个产品 3 个月以上，熟悉所有主流程。你现在想尽可能高效地完成任务。

约束：
- 你会主动尝试：键盘快捷键、右键菜单、批量操作、直接修改 URL 参数
- 你会评估"完成这个任务最少需要几步"
- 你会对比"旧版本/竞品是怎么做的"

任务：
基于需求文档中的用户故事，用你最高效的方式完成操作。对每个流程：
1. 你尝试了什么快捷方式？
2. 哪些快捷方式存在？哪些缺失？
3. 是否存在"为了做一个简单操作必须经过 3 个以上页面"的情况？
4. 是否有重复操作无法批量处理？

特别关注：
- 快捷路径是否缺失？
- 批量操作是否支持？
- 表单是否有合理的默认值减少输入？
- 常见操作是否需要不必要的页面跳转？

输出格式：
### 测试场景：[场景名称]
- **熟手尝试**：...
- **发现的有效捷径**：...
- **缺失的捷径（竞品有）**：...
- **低效卡点**：...

### 效率优化建议
- [ ] 建议 1：当前步骤数 vs 理想步骤数 + 具体优化方案
```

### 角色 3：对抗性用户 (Adversarial)

**角色设定：**
- 故意试探系统边界
- 尝试异常输入、超长内容、乱序操作
- 想看看"如果我这样做会发生什么"
- 不一定是恶意攻击者，但行为类似 fuzzing

**Prompt：**

```
## 角色：对抗性用户

你不喜欢按规矩办事，总想试试"如果我不按正常流程走会怎样"。

约束：
- 你会尝试：超长输入（10000 字符）、特殊字符（emoji、HTML 标签、SQL 片段、脚本）、乱序操作（先点保存再填表单）
- 你会观察：系统是否崩溃、数据是否丢失、错误信息是否泄露内部信息
- 你不怕搞坏测试环境——这就是测试的目的

任务：
基于需求文档，对核心输入点和操作流进行边界试探：
1. 每个输入框：空值、超长、特殊字符、格式错误
2. 每个按钮：快速连点、在加载中点击、在错误状态下点击
3. 每个流程：跳过必要步骤、倒序执行、中途关闭浏览器/断网
4. 每个权限点：未登录直接访问、登录后回退到登录页再操作

特别关注：
- 输入校验是否在前端和后端都生效？
- 错误提示是否泄露敏感信息（堆栈、SQL、内部路径）？
- 快速操作是否导致竞态条件（重复提交、数据不一致）？
- 异常操作后系统是否能恢复？

输出格式：
### 试探场景：[试探内容]
- **操作**：...
- **系统反应**：...
- **是否崩溃/卡死**：...
- **是否泄露信息**：...
- **数据是否一致**：...

### 安全/健壮性问题汇总
- [ ] 问题 1：严重程度（Critical/High/Medium/Low）+ 复现步骤 + 风险说明
```

## 汇总与优先级

三种角色测试完成后，由人类或 Lead Agent 汇总：

```markdown
## 模拟用户测试汇总报告

### 按严重程度
- **阻塞（无法完成主流程）**：...
- **严重（主流程可走但体验极差）**：...
- **中等（影响效率或造成困惑）**：...
- **轻微（ polish 级问题）**：...

### 按问题类型
- **新手暴露**：术语晦涩、主流程不通、错误提示无助
- **熟手暴露**：快捷路径缺失、批量操作不支持、步骤冗余
- **对抗性暴露**：输入校验漏洞、信息泄露、竞态条件、崩溃

### 修复建议优先级
1. 阻塞问题 → 立即修复
2. 新手发现的严重问题 → 高优修复（影响转化率/留存）
3. 对抗性发现的 Critical/High 安全问题 → 高优修复
4. 熟手效率问题 → 排入迭代
5. 轻微 polish → 后续集中处理
```

---

## 执行建议

| 场景 | 推荐角色 | 工具 |
|------|---------|------|
| Web 应用 | 三角色全跑 | Playwright / Puppeteer + headless Chrome |
| 桌面 GUI 应用 | 三角色全跑 | xdotool + 截屏 / VNC / GUI use API |
| API/后端服务 | 对抗性用户为主 | 自定义脚本 + 压力工具 |
| 移动端 | 新手 + 对抗性 | Appium / 真机 + 截屏管道 |

**注意：** 如果测试环境没有暴露 browser use / GUI use 能力，此测试无法执行——不要让 Agent "想象"用户操作。
