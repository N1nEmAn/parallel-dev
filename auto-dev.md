# auto-dev — TPDD 流程 + 工程纪律 Prompt 模板

本文件包含两部分：
1. **测试计划驱动开发 (TPDD)** —— 功能正确性保障
2. **工程纪律注入** —— 可维护性保障

开发 Agent 在执行任务时，应同时加载这两套规则。

---

## Part 1: TPDD — 测试计划驱动开发

### 核心流程

```
需求文档
    |
    v
[Phase 1] 生成测试计划（本 Agent）
    |
    v
[Phase 2] 独立审核测试计划（另一 Agent 或人类）
    |
    v
[Phase 3] 架构设计 + 开发实现（本 Agent）
    |
    v
[Phase 4] 运行测试 → 未通过 → 自主 debug → 修复 → 重跑（循环）
    |
    v
全绿 + 自审查通过 → 提交
```

### Phase 1: 生成测试计划 Prompt

```
你即将开始开发以下功能。但在写任何业务代码之前，你必须先产出一份完整的测试计划。

需求文档：
---
[粘贴结构化需求文档]
---

请产出一份覆盖三个层次的测试计划：

## 1. 单元测试（Unit Tests）
针对每个函数/类/模块的独立行为：
- 正常路径输入输出
- 边界值（空、最大、最小、刚好越界）
- 异常输入（null、undefined、类型错误、非法格式）
- 状态依赖（前置条件不满足时）
- 每个分支覆盖（if/else、switch、try-catch）

## 2. 集成测试（Integration Tests）
验证模块间协作：
- 模块 A 调用模块 B 的正常流程
- 模块 B 返回异常时模块 A 的处理
- 数据在模块间传递时的转换/验证
- 共享状态/资源的并发访问（如有）

## 3. 端到端测试（E2E Tests）
模拟真实用户完整流程：
- 主流程：从入口到出口的完整 happy path
- 分支流程：每个用户决策点的不同选择
- 异常流程：网络中断、服务不可用、超时、用户中途取消
- 数据一致性：操作完成后数据库/缓存/文件系统状态是否正确

### 输出格式
对每个测试用例，必须包含：
- **编号**：如 TC-UNIT-01
- **描述**：一句话说明测什么
- **前置条件**：执行前系统必须处于什么状态
- **输入**：具体输入值或动作
- **预期输出/行为**：明确的成功标准（不是"应该正常工作"，而是"返回 HTTP 200，body 包含字段 x=y"）
- **优先级**：P0（阻塞发布）/ P1（重要）/ P2（最好有）

### 规则
- 测试计划必须覆盖需求文档中所有"验收标准"
- 对需求文档中提到的每个边界条件，至少有一个测试用例
- 对需求文档中提到的每个错误场景，至少有一个测试用例
- 如果某个需求点无法被上述三类测试覆盖，明确标注出来并说明原因
```

### Phase 2: 独立审核 Prompt

**审核者必须是另一个全新会话的 Agent，或人类。** 禁止让生成测试计划的同一个 Agent 自己审自己。

```
你的任务是作为独立审核员，审查一份测试计划的质量。

需求文档：
---
[粘贴需求文档]
---

测试计划：
---
[粘贴测试计划]
---

请从以下维度审查：

1. **覆盖完整性**
   - 需求文档中的每个功能点是否都有对应测试？
   - 每个验收标准是否被验证？
   - 边界条件是否齐？
   - 错误路径和异常分支是否覆盖？

2. **测试有效性**
   - 每个测试用例的预期输出是否是可验证的？
   - 是否存在"测试在验证已经发生了什么"而非"应该发生什么"的自我循环？（陷阱：测试假设实现已经正确，只验证当前输出）
   - 测试用例之间是否独立？（一个失败不应导致另一个必然失败）

3. **业务盲区**
   - 从业务角度，是否遗漏了真实用户会遇到的场景？
   - 安全边界（权限、注入、越权）是否被考虑？
   - 性能和并发场景（如果需求中有约束）是否有测试？

输出格式：
- **通过**：列出确认高质量的测试用例编号
- **问题**：列出每个问题，标注严重程度（Critical / High / Medium / Low），给出修改建议
- **缺失**：列出遗漏的测试场景，说明为什么需要补

如果存在 Critical 或 High 级别问题，测试计划不能进入开发阶段。
```

### Phase 3-4: 开发与自测 Prompt

```
测试计划已经审核通过。现在开始开发。

需求文档：[链接或粘贴]
测试计划：[链接或粘贴]

开发规则：
1. 先基于工程纪律（见 Part 2）做架构设计和模块划分
2. 按测试计划中的优先级实现：先 P0，再 P1，再 P2
3. 每完成一个模块，立即编写对应的测试代码并运行
4. 测试失败时，使用调试工具（日志、断点、堆栈跟踪）定位问题，不要靠猜测
5. 所有 P0 测试必须通过；P1 未通过需记录原因；P2 可作为后续迭代
6. 开发完成后，运行完整测试套件，确保没有回归

环境要求：
- [列出可用的 shell / 测试框架 / 浏览器 / 调试器 / 其他工具]
- 如果遇到环境缺失，先尝试自行安装配置；无法解决时上报

调试原则：
- 先看日志和错误信息，不是直接改代码试错
- 复现问题后再修复；不能稳定复现的 bug 要记录复现条件
- 修复后不仅要让当前测试通过，还要思考是否引入了新的边界问题
```

## Part 2: 工程纪律 —— 八大原则

Agent 在编码、架构设计、自审查时，必须逐条对照以下原则。

### 原则详解

#### 1. 模块深度 (Deep Modules)

好的模块对外接口简单、内部功能深厚。Agent 天然倾向于过度拆分——把类切得很小很细，形成一堆浅层模块。

**检查清单：**
- [ ] 这个模块的接口是否比它的实现简单得多？
- [ ] 是否存在"一个函数只做一行事，然后调下一个函数"的浅层链？
- [ ] 禁止为拆而拆：如果一个模块的接口和实现复杂度差不多，考虑合并

**反例（需避免）：**
```python
# 浅模块：接口没比实现简单
def get_user_name(user):
    return user.name

def get_user_email(user):
    return user.email
# → 应该合并为一个 get_user_profile(user) -> {name, email}
```

#### 2. 信息隐藏 (Information Hiding)

模块之间不应该共享各自的内部实现知识。按"谁拥有这个知识"拆分，不是按执行顺序拆分。

**检查清单：**
- [ ] 修改模块 A 的内部实现，是否需要改模块 B 的代码？如果是，说明信息泄露
- [ ] 是否存在"先做 A 再做 B，所以拆成两个模块"的时间顺序拆分？
- [ ] 模块的接口是否暴露了内部数据结构？

**反例：**
```python
# 模块 B 知道模块 A 内部用列表存储
for item in module_a.get_internal_list():  # 泄露！
    process(item)
# → 应该 module_a.iter_items(callback) 或返回抽象迭代器
```

#### 3. 抽象分层 (Layered Abstraction)

每层提供不同的思维模型。禁止只把调用原封不动转给下一层的"透传层"。复杂度往下压。

**检查清单：**
- [ ] 每一层是否提供了比下一层更高级的概念？
- [ ] 是否存在"函数 A 接收参数，直接传给函数 B，函数 B 直接传给函数 C"的透传链？
- [ ] 上层代码是否仍然在处理本应由下层处理的细节？

**反例：**
```python
# 透传层：没有存在的意义
def service_create_user(name, email):
    return db_create_user(name, email)  # 没有新增抽象
```

#### 4. 内聚与分离 (Cohesion & Separation)

必须同时理解才有意义的代码应该在一起；通用逻辑和特例逻辑混在一起导致两者都难理解的，就该分开。

**检查清单：**
- [ ] 这个函数是否包含了两种不同抽象层次的逻辑？
- [ ] 是否存在"不读完另一个函数就无法理解当前函数"的耦合？
- [ ] 通用工具和业务特例是否混在一起？

#### 5. 错误处理 (Error Handling)

异常的泛滥是复杂度的隐形杀手。尽量"把错误定义掉"。

**检查清单：**
- [ ] 能否通过调整语义或默认行为，让某些"错误"不再发生？
- [ ] 是否存在到处 try-catch 但只是打印日志或重新抛出的模式？
- [ ] 错误处理路径是否比正常路径还复杂？如果是，考虑重新设计

**好例子：**
```python
# 把错误定义掉：不存在"找不到"的情况，返回空列表是正常行为
def list_comments(post_id) -> List[Comment]:
    return db.query(...).all()  # 没有记录 → 空列表，不是异常
```

#### 6. 命名与显而易见性 (Naming & Obviousness)

读者第一眼就能正确猜出代码在做什么。命名精确、全库一致、不反直觉。

**检查清单：**
- [ ] 新加入项目的开发者读到这段代码，会"感到惊讶"吗？
- [ ] 同名变量/函数在不同模块中含义是否一致？
- [ ] 缩写是否是团队公认的标准？避免个人缩写
- [ ] 布尔变量命名是否清晰？（is_valid, has_permission, should_retry）

#### 7. 文档与注释 (Documentation)

注释描述代码本身无法表达的东西：设计意图、为什么选这个方案、接口的抽象语义。

**检查清单：**
- [ ] 注释是否只是在重复代码已经说清楚的？→ 删除
- [ ] 是否存在"为什么这样写"的隐含决策没有注释？→ 补充
- [ ] 接口的抽象语义是否明确？（"做什么"而不是"怎么做"）

**好注释 vs 坏注释：**
```python
# 坏：重复代码
i += 1  # increment i by 1

# 好：解释意图和约束
# 使用乐观锁：version 字段在更新时自动递增，
# 如果读取后的 version 已变，说明并发冲突，需重试
updated = db.update(..., where_version=old_version)
```

#### 8. 战略式设计 (Strategic Design)

每次修改都是对整体设计的一次投资，而非战术性补丁。抵制"先让它跑起来再说"，同时警惕过度设计。

**检查清单：**
- [ ] 这次修改是否在增加技术债？有没有更干净的方式多花 20% 时间但能减少未来 200% 的维护成本？
- [ ] 是否在解决"假想的需求"？只解决眼前的问题
- [ ] 新增的抽象是否真的有多个调用方？如果只有一个，YAGNI

## Part 3: 自审查清单

开发完成、测试全绿后，Agent 必须对照以下清单进行一轮自审查：

```markdown
## 自审查报告

### 架构层面
- [ ] 模块划分是否合理？是否存在过浅/过深的模块？
- [ ] 模块间接口是否稳定？修改一个模块的内部实现是否会影响其他模块？
- [ ] 是否存在只转发不做事的中间层？
- [ ] 复杂度是否主要压在底层？上层代码是否保持简洁？

### 代码质量
- [ ] 命名是否精确、一致、不反直觉？
- [ ] 每个函数是否只做一件事？（如果函数名里出现了"and"，警惕）
- [ ] 是否存在重复代码？能否提取共用逻辑？
- [ ] 错误处理是否简洁？是否尽量"把错误定义掉"？

### 测试层面
- [ ] 测试是否覆盖了所有 P0 场景？
- [ ] 测试是否在验证"应该发生什么"而非"已经发生了什么"？
- [ ] 测试代码本身是否好懂？测试名是否描述了行为和预期？
- [ ] 是否存在脆弱测试（依赖执行顺序、依赖具体时间、依赖随机数未 mock）？

### 可维护性
- [ ] 新加入的开发者能否在 5 分钟内理解核心流程？
- [ ] 修改一个功能是否需要改多个不相关的地方？
- [ ] 是否引入了新的外部依赖？是否必需？

### 发现的问题
[列出发现的问题及修复措施；无法自行修复的标注原因]
```

## 非功能需求 Skill 扩展

当项目需要覆盖功能测试之外的特殊领域时，为 Agent 准备专门的 Skill 文件：

### 压力测试 Skill 要点
- 构造高并发负载的方式（k6 / locust / wrk）
- 观测指标：p50 / p99 / p999 延迟、错误率、吞吐量曲线
- 退化曲线识别：吞吐量随并发数增加是否线性？何时出现拐点？
- SLA 破线定义：什么指标超过什么阈值算失败

### 混沌测试 Skill 要点
- 故障注入对象：数据库、下游服务、网络、磁盘
- 故障类型：kill、延迟、丢包、分区、磁盘满
- 验证目标：降级策略、恢复时间、数据一致性

### 安全测试 Skill 要点
- 常见漏洞扫描清单：XSS、SQLi、权限越权、CSRF、SSRF、路径遍历
- 输入校验边界：超长字段、特殊字符、编码绕过、空字节注入
- 认证授权：未认证访问、水平越权、垂直越权

### 回归演进测试 Skill 要点
- 架构劣化指标：文件行数、函数复杂度（cyclomatic）、模块间依赖环
- 定期跑（如每周 CI）而非每次提交
- 超阈告警而非直接失败
