# auto-triage — 产出消化框架（实验性）

> **状态：设计预留，当前未跑通。**
>
> 这是并行开发闭环的第四块积木：Agent 产出 → Agent 自动消化整理 → 新工作项派回给 Agent。
> 今天这个闭环最薄弱，但值得持续投入。

## 问题定义

当你同时运行 3+ 个 Agent 时，第一周觉得节奏棒极了，第二周就会发现——你的一天大半时间用来读各种报告，其中不少是重复的、或低优先级小问题被堆到你面前。

你从"审代码"变成了"读各种报告"。瓶颈悄悄回到了你身上。

## 理想闭环

```
你下发需求
    |
    v
Agent 并行开发
    |
    v
Agent 自动消化产出
    - 批量读完找规律和重复
    - 合并同一件事的不同描述
    - 按类型派给下一轮开发
    - 按"影响多少用户 × 有没有临时办法"打优先级
    - 每条附一句理由
    |
    v
只有真正需要你拍板的高优问题升级上来
    |
    v
新的工作项派回给 Agent
```

## 当前限制（为什么还没跑通）

| 环节 | 问题 |
|------|------|
| 分类 | 容易失焦，把不同根因的问题归为一类 |
| 合并 | 容易漏真正独立的问题，把"症状相似但根因不同"的合并掉 |
| 优先级判断 | 缺业务视角，无法准确评估"影响多少用户" |
| 行动建议 | 推荐的"建议动作"有时不切实际 |

## 过渡期缓冲措施（今天就能做）

在闭环跑通之前，用以下方式降低你的阅读负担：

### 1. 强制结构化产出模板

每个 Agent 的报告必须按以下格式输出：

```markdown
## 产出摘要
- **任务**：一句话
- **状态**：完成 / 部分完成 / 阻塞
- **关键结果**：最多 3 条 bullet

## 详细报告（如有）
[链接到详细文件，不直接贴在消息里]

## 需要人类决策的事项
| 事项 | 严重程度 | 建议动作 | 不做的风险 |
|------|---------|---------|-----------|
| ...  | 阻塞/高/中/低 | ... | ... |

## 无行动需求的信息
[纯告知性质的内容，人类不需要响应]
```

### 2. 高频低优问题写进 Skill 让 Agent 自己忽略

建立一份 "auto-ignore" 规则：
- 警告级别低于某阈值的 linter 问题 → Agent 自行决定是否修复
- 已知环境问题（如测试数据库偶发延迟）→ 标注"已知，不升级"
- 文档拼写错误（非 API 文档）→ 批量攒到一定数量再处理

### 3. 固定"读产出"时间块

不要让 Agent 的输出像聊天消息一样碎片化切割你的整天。

建议节奏：
- 上午 10:00 和下午 16:00 各留 30 分钟集中读产出
- 非紧急阻塞问题不立即响应，攒到时间块统一处理
- 紧急阻塞问题通过特定关键词触发即时通知（如 "[BLOCKING]"）

## 未来设计方向

当 auto-triage 能力成熟后，预期行为：

### 输入
- 多个 Agent 的产出报告（代码、测试结果、用户测试反馈、复盘总结）
- 项目上下文（优先级、里程碑、已知问题库）

### 处理
1. **去重合并**：识别"同一问题的不同描述"，合并为单条
2. **根因归类**：按技术领域（前端/后端/infra/安全）和类型（bug/性能/债务/需求澄清）分类
3. **优先级打分**：
   - 影响面：多少用户 / 多少功能受影响
   - 紧急度：是否有临时办法 / 是否阻塞发布
   - 修复成本：预估工作量
   - 综合得分 = f(影响面, 紧急度, 修复成本)
4. **动作路由**：
   - 自动修复类（低优 linter、文档 typo）→ 直接派给 Agent
   - 需确认类（设计决策、资源申请）→ 升级给人类
   - 需调查类（偶发 bug、性能退化）→ 派给 debugger Agent
   - 需排期类（大重构、技术债）→ 加入 backlog

### 输出
- 人类收件箱：只有需确认的事项，每条带上下文和推荐选择
- Agent 任务队列：自动修复和调查任务直接派单
- 聚合看板：今日产出概览、趋势图（bug 增长率、测试覆盖率变化）

## 与现有 Skill 的衔接

| 上游产出 | 当前消化方式 | 未来 triage 方式 |
|---------|------------|----------------|
| auto-req 需求文档 | 人类审 | 让另一个 Agent 审结构完整性，人类只审业务 |
| auto-dev 测试计划 | 人类或另一 Agent 审 | triage Agent 先过滤格式问题，人类审业务盲区 |
| auto-dev 测试报告 | 人类看哪些红了 | triage Agent 分类失败原因，自动给 debugger 派单 |
| auto-test 用户反馈 | 人类读三份报告 | triage Agent 按严重程度汇总，去重相似问题 |
| 代码提交 / PR | 人类 review | 低优改动让 triage Agent 过一遍，高优升级人类 |

## 暂不实现，但值得记录的想法

- **时间衰减**：同样 severity 的问题，反复出现第 3 次自动升级优先级
- **关联推荐**："你正在看问题 A，相关的问题 B 和 C 可能也值得关注"
- **模式识别**："过去 3 个迭代中，模块 X 的 bug 密度是其他模块的 5 倍，建议安排重构"
